# Chapter 4: rpcgen — The Protocol Compiler

## What rpcgen Does

rpcgen reads a `.x` file (a protocol definition written in a language called **RPCL**) and generates C code that implements the protocol.

It's like `protoc` for Protocol Buffers, or `thrift` for Apache Thrift. You write the schema, rpcgen writes the marshalling code. You focus on your logic, not on packed byte manipulation.

## What a .x File Looks Like

A `.x` file declares types and procedures:

```rpc
/* calculator.x — Our calculator protocol */
program CALC_PROG {
    version CALC_VERS {
        int ADD(int)    = 1;   /* procedure 1: add two ints */
        int SUB(int)    = 2;   /* procedure 2: subtract */
        int MUL(int)    = 3;   /* procedure 3: multiply */
        int DIV(int)    = 4;   /* procedure 4: divide */
    } = 1;
} = 400001;
```

Wait — those procedure signatures look wrong. `int ADD(int)` takes one int and returns one int. But we want to pass two ints (a and b). Let me fix that:

```rpc
/* calculator.x — Correct version */
struct calc_args {
    int a;
    int b;
};

struct calc_result {
    int result;
    int error;       /* 0 = OK, 1 = division by zero, etc. */
};

program CALC_PROG {
    version CALC_VERS {
        calc_result ADD(calc_args)  = 1;
        calc_result SUB(calc_args)  = 2;
        calc_result MUL(calc_args)  = 3;
        calc_result DIV(calc_args)  = 4;
    } = 1;
} = 400001;
```

Each procedure has: a return type, a name, an argument type, and a procedure number. The argument type is a struct (or a base type if there's only one parameter). The return type is a struct (or void if there's no meaningful return).

## What rpcgen Generates

When you run `rpcgen calculator.x`, you get these files:

### calculator.h — The Protocol Header

```c
/*
 * Generated by rpcgen. Do not edit.
 */
struct calc_args {
    int a;
    int b;
};
typedef struct calc_args calc_args;

struct calc_result {
    int result;
    int error;
};
typedef struct calc_result calc_result;

#define CALC_PROG 400001
#define CALC_VERS 1
#define ADD 1
#define SUB 2
#define MUL 3
#define DIV 4
```

This header defines your types and procedure numbers. It's the contract. Your client and server code both include it.

### calculator_xdr.c — The Serialization Code

```c
/* Generated XDR encode/decode */
bool_t xdr_calc_args(XDR *xdr, calc_args *args) {
    return (xdr_int(xdr, &args->a) &&
            xdr_int(xdr, &args->b));
}

bool_t xdr_calc_result(XDR *xdr, calc_result *res) {
    return (xdr_int(xdr, &res->result) &&
            xdr_int(xdr, &res->error));
}
```

This is the marshalling code. `xdr_calc_args` reads or writes the struct fields in order, depending on the XDR operation direction (`XDR_ENCODE` or `XDR_DECODE`). The generated code uses the user-space XDR library (`libtirpc`), which is different from the kernel's `xdr_stream`.

### calculator_clnt.c — The Client Stub

```c
/* Generated client code (simplified) */
calc_result *calc_add_1(calc_args *args, CLIENT *clnt) {
    calc_result *res;
    /* Serialize args, send RPC, deserialize result */
    if (clnt_call(clnt, ADD, xdr_calc_args, args, xdr_calc_result, &res, timeout))
        return NULL;
    return res;
}
```

This is the function you call from your application. It marshals the arguments, calls `clnt_call()` (the user-space RPC function), and unmarshals the result. You don't need to know how the serialization works — you just call `add(3, 4)` and get a result back.

### calculator_svc.c — The Server Stub

```c
/* Generated server code */
static void calc_add_1_svc(calc_args *args, calc_result *res, struct svc_req *rqstp) {
    res->result = args->a + args->b;
    res->error = 0;
}
```

This is the function you fill in with your logic. rpcgen generates the dispatch infrastructure; you write the actual computation.

## The rpcgen Command-Line Flags

| Flag | What It Does |
|------|-------------|
| (no flags) | Generate all files (.h, _xdr.c, _clnt.c, _svc.c) |
| `-a` | Generate all files plus a sample Makefile and client/server templates |
| `-c` | Generate _xdr.c only |
| `-h` | Generate .h only |
| `-l` | Generate _clnt.c only |
| `-m` | Generate _svc.c only |
| `-t` | Generate type definitions only (no RPC stubs) |
| `-s transport` | Generate server code for a specific transport (tcp, udp, etc.) |
| `-o output` | Write to a specific output file |
| `-I` | Include protocol header in generated files |
| `-C` | Generate ANSI C code (default) |
| `-K seconds` | Set server idle timeout |
| `-D name[=value]` | Define a preprocessor macro |
| `-U name` | Undefine a preprocessor macro |

The most useful flags for day-to-day work:

```bash
# Generate everything: header, XDR, client stub, server stub
rpcgen calculator.x

# Generate just the header (for documentation/sharing)
rpcgen -h calculator.x

# Generate just the XDR code (if you're writing stubs by hand)
rpcgen -c calculator.x
```

## "Why Can't I Use This in the Kernel?"

Here's the problem: rpcgen generates code that depends on **libtirpc** — the user-space RPC library. It uses `CLIENT *` handles, `clnt_call()`, `SVCXPRT *`, and other user-space constructs that don't exist in the kernel.

The kernel has its own RPC infrastructure (`sunrpc.ko`), and it works differently:

| Aspect | User-space (rpcgen) | Kernel (sunrpc.ko) |
|--------|-------------------|-------------------|
| Client handle | `CLIENT *` | `struct rpc_clnt *` |
| Server handle | `SVCXPRT *` | `struct svc_xprt *` |
| Call function | `clnt_call()` | `rpc_run_task()` |
| XDR type | `XDR` struct + `xdr_int()` etc. | `struct xdr_stream` + `xdr_reserve_space()` |
| Auth | `auth_destroy()` | `rpc_auth` structures |
| Dispatch | Generated `_svc.c` | Manual via `svc_program_register()` |

The kernel's XDR layer is different from the user-space XDR library. Functions like `xdr_int()` don't exist in the kernel. The kernel uses `xdr_reserve_space()` + manual byte manipulation.

**But rpcgen is still useful.** The `.x` file documents your protocol in a machine-readable format. Anyone who builds a new client or server implementation can run rpcgen to get a reference implementation. The header file (`calculator.h`) is usable in the kernel — it's just struct definitions and constants.

In practice, you:

1. **Write the .x file** to document your protocol
2. **Run `rpcgen -h`** to generate the header with struct definitions and constants
3. **Write the XDR code by hand** for the kernel, using `xdr_stream`
4. **Write the client/server logic by hand** using `rpc_clnt` / `rpc_run_task`

The .x file is the source of truth. The kernel code implements the same protocol, but through the kernel's own infrastructure.

## Do You Need rpcgen?

For kernel development: **maybe not**. If you know your protocol — the structs, the procedure numbers, the wire format — you can write the XDR encode/decode functions directly. The .x file is documentation, not code.

For any serious protocol where multiple people need to understand the wire format: **yes**. The .x file is the single source of truth. It prevents ambiguity. When someone asks "what's the wire format for the error field in calc_result?", the answer is "read calculator.x."
